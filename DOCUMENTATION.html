Overview

This is the Microsoft Agent Framework (MAF) rewrite of the SQL Server Blocking Agent. It does exactly the same job as the LangGraph version in the companion repo (sql-blocking-agent/), but uses a completely different technology stack — zero LangChain, zero LangGraph.

💬 What Is This? (Plain English)

Imagine a Security Guard for Your Database

A SQL Server database handles many users and applications at the same time. Sometimes one user or program holds a "lock" on data — like holding a file open — and everyone else has to wait. If this goes on too long, your application grinds to a halt. We call this blocking.

This agent is an automated security guard. Every 45 seconds it checks: "Is anyone being blocked right now?" If yes, it investigates — it reads dozens of database metrics, sends everything to an AI (GPT-4.1), and asks: "Is this serious enough to kill the offending connection?"

Before it does anything dangerous, it checks 8 safety rules written in code that the AI cannot override. Only if all rules pass does it pull the trigger. After killing the blocker, it writes a permanent audit record and generates a root-cause analysis report explaining why the blocking happened and how to prevent it.

It runs as a Microsoft Agent Framework workflow — a sequence of steps (called executors) where each step either queries the database, calls the AI, or writes results. It replaces the LangGraph version without changing any of the underlying SQL logic.

▶️ Entry Point — main.py

Plain English: This is where the agent starts. Python runs main(), which reads all configuration, then enters an infinite loop — checking for blocking every 45 seconds (adjustable). Each check is called a cycle.

What happens on startup

  1. Load .env file — reads API keys and server names from a config file
  2. load_config() — reads additional settings from SQL Server's AgentConfigDB table, stores them in a global singleton so all code can access them without passing parameters around
  3. Enter polling loop — runs one cycle, sleeps, repeats forever until Ctrl+C
# main.py — entry point
import asyncio, sys, os, uuid
from pathlib import Path
from dotenv import load_dotenv

# Load .env BEFORE importing agents (they need API keys at import time)
load_dotenv()

# Add src/ to Python's search path so "from orchestrator.x import y" works
sys.path.insert(0, str(Path(__file__).parent / "src"))

from orchestrator.config   import load_config     # reads SQL Server config table
from orchestrator.state    import BlockingState    # Pydantic model for cycle data
from orchestrator.workflow import AGENT_WORKFLOW   # the compiled MAF workflow

async def run_cycle(config: dict) -> BlockingState:
    initial = BlockingState(
        server_name=config["server_name"],   # which SQL Server to monitor
        correlation_id=str(uuid.uuid4())[:8], # unique ID for log correlation
        dry_run=config["dry_run"],            # True = never actually KILL
        cycle_start_utc=datetime.now(timezone.utc).isoformat(),
    )
    # Run all executors in sequence — returns the final BlockingState
    # from notification_node.yield_output()
    return await AGENT_WORKFLOW.run(initial)

async def main():
    config = load_config()       # sets _ACTIVE_CONFIG singleton
    poll_seconds = config["poll_interval_seconds"]  # default: 45s
    while True:
        try:
            await run_cycle(config)
        except Exception as e:
            log.error("Cycle crashed: %s", e)     # log but don't stop
        await asyncio.sleep(poll_seconds)

asyncio.run(main())

🔗 WorkflowBuilder — Wiring the Pipeline

Plain English: WorkflowBuilder is like a railway track layout. You specify which station (executor) comes after which, and optionally add a switch (FanOutEdgeGroup) so the train can go two different ways depending on what it finds. builder.build() finalises the track — it cannot be changed after that.

This replaces LangGraph's StateGraph + add_conditional_edges + compile().

# orchestrator/workflow.py
from agent_framework import WorkflowBuilder, FanOutEdgeGroup

def build_workflow():
    # Start the builder — detection is always the first executor
    builder = WorkflowBuilder(start_executor=detection_node)

    # After detection: go to analyzer (blocking found) OR notification (clean)
    builder.add_edge_group(FanOutEdgeGroup(
        source_id="detection",
        target_ids=["analyzer", "notification"],
        selection_func=route_after_detection,   # returns list[str]
    ))

    # analyzer always flows to determination
    builder.add_edge(analyzer_node, determination_node)

    # After determination: go to action (KILL) OR rca (ALERT/SKIP)
    builder.add_edge_group(FanOutEdgeGroup(
        source_id="determination",
        target_ids=["action", "rca"],
        selection_func=route_after_determination,
    ))

    # action → rca → notification (linear tail)
    builder.add_edge(action_node, rca_node)
    builder.add_edge(rca_node, notification_node)

    return builder.build()           # compiles — immutable from here

AGENT_WORKFLOW = build_workflow()   # created once at import time

Routing Functions

These are the "switch controllers" — called by FanOutEdgeGroup after each conditional node. They return a list of strings (executor IDs to run next):

# orchestrator/router.py
def route_after_detection(state: BlockingState) -> list[str]:
    return ["analyzer"] if state.get("has_blocking") else ["notification"]

def route_after_determination(state: BlockingState) -> list[str]:
    return ["action"] if state.get("decision") == "KILL" else ["rca"]

Key difference from LangGraph

LangGraph routing returns a single str. MAF routing returns list[str] to support true fan-out (running multiple executors in parallel). In this graph we always return a list with one item, but the API supports real parallelism.

🔧 The agent_framework Shim

Plain English: Microsoft Agent Framework 1.0 is a new SDK. To make this code runnable immediately without needing preview access, we built a compatibility shim — a small Python package (src/agent_framework/) that implements the MAF API on top of the standard openai library. Think of it as a translation layer.

WorkflowContext

Passed to every executor. send_message(state) forwards state to the next executor. yield_output(state) marks the terminal node — the workflow stops and returns this state.

@executor(id="...")

A decorator that tags an async def function with an executor ID. WorkflowBuilder uses this ID to wire up edges.

FoundryChatClient

Wraps openai.OpenAI to accept a project_endpoint and a credential object. Handles both API key auth and managed identity auth.

ChatAgent

A thin async wrapper: takes a FoundryChatClient and a system prompt, exposes await agent.run(user_message) returning a result with a .content string.

# src/agent_framework/__init__.py — simplified view

class WorkflowContext:
    async def send_message(self, state):
        self._next_state = state       # forward to next executor
    async def yield_output(self, state):
        self._output = state           # stop workflow, return this
        self._terminal = True

def executor(id: str):
    def decorator(fn):
        fn._executor_id = id           # tags the function
        return fn
    return decorator

class Workflow:
    async def run(self, initial_state):
        state = initial_state
        current_id = self._start_id
        while current_id:
            fn = self._executors[current_id]
            ctx = WorkflowContext()
            await fn(state, ctx)          # call the executor
            if ctx._terminal:
                return ctx._output        # notification_node called yield_output
            state = ctx._next_state
            current_id = self._next_ids(current_id, state)[0]
        return state

📦 BlockingState — The Shared Data Object

Plain English: BlockingState is like a clipboard that gets passed from one executor to the next. Each executor reads what it needs, writes its results, and passes it on. By the time it reaches the last executor, it contains everything that happened during the cycle.

Unlike LangGraph (which uses a plain dict), MAF uses a Pydantic BaseModel — a typed Python object with validated fields. Executors write directly to attributes: state.has_blocking = True instead of returning {"has_blocking": True}.

🔍 Executor 1 — detection_node

Tool Executor — No LLM

What it does

Plain English: This is the agent's eyes. It runs 6 different SQL queries against the database's internal monitoring tables (called DMVs — Dynamic Management Views) to build a complete picture of any blocking. No AI is involved — it's pure, fast database queries.

It looks at: who is blocking whom, how long they've been waiting, what lock they're holding, whether the transaction log is nearly full, whether the blocker's query has a bad execution plan, and what category of blocking this is.

If nothing is blocking, it short-circuits to notification immediately. This keeps the cycle cheap — no LLM cost on a clean system.

The 6 tools it calls (in order):

  1. detect_blocking() — scans sys.dm_exec_requests to find sessions blocking others. Identifies the head blocker (the root cause), all victims, wait times, and the full blocking chain.
  2. analyze_log_safety() — checks the transaction log fullness. If it's >80%, killing a session could trigger a large rollback that fills the log entirely. This feeds into governance rule R9.
  3. analyze_locks() — reads sys.dm_tran_locks to determine what type of lock is held (shared, exclusive, update), what object/row it's on, and whether lock escalation occurred.
  4. analyze_plan_cache() — retrieves the execution plan XML for the blocking query. Checks for parallelism (CXPACKET waits), parameter sniffing issues, and missing indexes.
  5. analyze_query_store() — checks Query Store for the query's execution history. Did performance suddenly get worse? Is there a better plan available?
  6. classify_scenario() — classifies the blocking into one of 7 KB (Knowledge Base) scenarios, e.g. scenario 6 = "idle session with open transaction" (the classic case where a developer forgot to commit).
# agents/detector/agent.py
@executor(id="detection")                # ← MAF decorator — gives this fn an ID
async def detection_node(state: BlockingState, context: WorkflowContext) -> None:
    cfg = get_config()                    # reads _ACTIVE_CONFIG singleton
    agent = BaseAgent(cfg)

    # Tool 1: detect blocking chains
    result = detect_blocking(DetectionInput(
        monitor_conn_str=agent.monitor_conn_str,
        kill_threshold_ms=cfg["kill_threshold_ms"],
    ), query_sql=agent.query_sql)

    state.has_blocking  = result.has_blocking
    state.blocking_rows = result.blocking_rows
    state.head_blocker  = result.head_blocker

    if not state.has_blocking:
        await context.send_message(state)   # route_after_detection → "notification"
        return

    # Tools 2–6: only run if blocking was found
    # (saves time on clean cycles)
    log_result = analyze_log_safety(...)
    state.log_used_pct = log_result.log_used_pct

    lock_result = analyze_locks(...)
    state.lock_type = lock_result.lock_type

    plan_result = analyze_plan_cache(...)
    state.plan_cache_hit = plan_result.plan_cache_hit

    qs_result = analyze_query_store(...)
    state.qs_better_plan_exists = qs_result.better_plan_exists

    scenario = classify_scenario(...)
    state.scenario_id = scenario.scenario_id

    await context.send_message(state)      # route_after_detection → "analyzer"

🧠 Executor 2 — analyzer_node

LLM Executor — GPT-4.1

What it does

Plain English: The Analyzer is the agent's first consultation with the AI. It takes all the raw numbers from the detection step — wait times, lock modes, plan XML, scenario type — and asks GPT-4.1 to synthesize them into a human-readable explanation.

Think of it as the AI writing a paragraph: "Session 99 has been holding an exclusive lock on dbo.BlockingTest for 44 seconds, blocking svc_appaccount. The query is an idle transaction (scenario 6). The plan shows no parallelism issues. Severity: HIGH."

This analysis is then passed to the Determination node to inform the kill decision. The AI doesn't decide whether to kill here — it only diagnoses.

The LLM call uses await self.ask_llm_json() — an async coroutine. This is the key MAF difference: all LLM calls are non-blocking async operations.

# agents/analyzer/agent.py
class AnalyzerAgent(BaseAgent):
    async def run(self, state: BlockingState) -> dict:
        user_msg = build_user_message(state)          # assembles JSON diagnostic data
        raw = await self.ask_llm_json(SYSTEM_PROMPT, user_msg)
        result = AnalyzerLLMResult(**raw)             # validated Pydantic model
        return {"llm_analysis": result.analysis,
                "severity_hint": result.severity,
                "diagnosis": result.diagnosis}

@executor(id="analyzer")
async def analyzer_node(state: BlockingState, context: WorkflowContext) -> None:
    cfg = get_config()
    updates = await AnalyzerAgent(cfg).run(state)
    state.llm_analysis = updates["llm_analysis"]   # AI narrative
    state.severity_hint = updates["severity_hint"]  # LOW/MEDIUM/HIGH/CRITICAL
    state.diagnosis     = updates["diagnosis"]       # structured findings
    await context.send_message(state)               # always goes to determination

async vs sync — the MAF difference

In LangGraph, ask_llm() was synchronous. In MAF, it is async def ask_llm() — an awaitable coroutine. The await keyword means Python can do other work while waiting for the GPT API to respond, making the system more efficient.

⚖️ Executor 3 — determination_node

LLM Executor + Hard Gates

What it does

Plain English: The Determination node is the judge. It makes the final KILL / ALERT_ONLY / SKIP decision. It works in two steps:

Step 1 (Code rules): Before asking the AI anything, it checks 8 hard-coded safety rules. These are like laws — if any rule matches, the decision is forced and the AI never gets a say. This prevents the AI from making reckless decisions.

Step 2 (AI decision): If no hard rule fired, the AI gets the full picture — analysis, diagnostics, scenario, log state — and decides: Is this worth killing? Is the risk acceptable?

The result (KILL, ALERT_ONLY, or SKIP) plus the reason are stored in state, and the router sends the workflow to action (if KILL) or rca (if not).

@executor(id="determination")
async def determination_node(state: BlockingState, context: WorkflowContext) -> None:
    cfg = get_config()
    updates = await DeterminationAgent(cfg).run(state)
    state.decision            = updates["decision"]          # KILL / ALERT_ONLY / SKIP
    state.risk_level          = updates["risk_level"]
    state.rule_triggered      = updates["rule_triggered"]     # 0 = LLM decided
    state.decision_reason     = updates["decision_reason"]
    state.dba_approval_required = updates["dba_approval_required"]
    await context.send_message(state)
    # → route_after_determination → ["action"] or ["rca"]

class DeterminationAgent(BaseAgent):
    async def run(self, state: BlockingState) -> dict:
        head = state.get("head_blocker") or {}
        spid = head.get("session_id")

        # ── Step 1: hard gates — LLM cannot override these ──
        gate = self._check_hard_gates(state, head, spid)
        if gate:                               # a rule matched
            decision, risk, reason, rule, dba = gate
            return self._build_result(decision, risk, reason, rule, dba)

        # ── Step 2: ask the LLM ──
        raw = await self.ask_llm_json(SYSTEM_PROMPT, user_msg)
        return DeterminationLLMResult(**raw)

⚡ Executor 4 — action_node

Tool Executor — No LLM

What it does

Plain English: The Action node does the actual work. It only runs when the Determination node decided KILL. It is pure code — no AI involved.

Before pulling the trigger, it does one more safety check: it re-queries the session to confirm it still exists and still belongs to the same user account (SPID recycling protection — a security measure to prevent accidentally killing an innocent session that reused the same session ID).

If dry_run=True (the default), it logs "would have killed SPID X" but doesn't actually run the KILL command. This is the training wheels mode for safe testing.

If dry_run=False, it issues KILL X, then waits and checks that the session is truly gone (or rolling back). Finally, it writes a permanent record to KillAuditLog.

@executor(id="action")
async def action_node(state: BlockingState, context: WorkflowContext) -> None:
    cfg = get_config()
    agent = BaseAgent(cfg)
    head = state.get("head_blocker") or {}

    # execute_kill: re-verifies SPID, then issues KILL (or simulates)
    exec_result = execute_kill(SqlExecutorInput(
        monitor_conn_str=agent.monitor_conn_str,
        session_id=head["session_id"],
        login_name=head["login_name"],
        dry_run=cfg["dry_run"],            # True = simulate only
    ), query_sql=agent.query_sql)

    # validate_kill: confirms SPID is gone (SPID recycling protection)
    val_result = validate_kill(SqlValidatorInput(
        monitor_conn_str=agent.monitor_conn_str,
        session_id=head["session_id"],
        expected_login=head.get("login_name", ""),  # Gap A: confirm same login
    ), query_sql=agent.query_sql)

    state.kill_executed = exec_result.kill_issued
    state.killed_spid   = exec_result.killed_spid
    state.kill_status   = val_result.kill_status    # SUCCESS / ROLLING_BACK / etc
    state.kill_time_utc = exec_result.kill_time_utc

    await context.send_message(state)              # always goes to rca_node

📋 Executor 5 — rca_node

LLM Executor

What it does

Plain English: RCA stands for Root Cause Analysis. After the blocking is handled (killed or alerted), the agent asks GPT-4.1 one more question: "Why did this happen, and how can we prevent it?" It looks at the query's execution plan, the blocking scenario, and historical records (has this same user or query blocked before?) to generate actionable recommendations.

The result is a structured report: what caused the blocking, how often it recurs, and concrete fixes (add an index here, change this isolation level, reduce this connection pool size, add a MAXDOP hint).

⚠️ The Async/Sync Bridge

The tools/rca.py tool takes a synchronous ask_llm_json callable (a plain function, not a coroutine). But MAF's ask_llm_json is async. The bridge: ask_llm_json_sync() in BaseAgent runs the async code in a dedicated thread using ThreadPoolExecutor + asyncio.run(), preventing "nested event loop" errors.

@executor(id="rca")
async def rca_node(state: BlockingState, context: WorkflowContext) -> None:
    cfg = get_config()
    agent = BaseAgent(cfg)

    rca_input = RCAInput(
        head_blocker=state.head_blocker,
        blocking_rows=state.blocking_rows,
        scenario_id=state.scenario_id,
        llm_analysis=state.llm_analysis,
        # ... other fields ...
    )

    # generate_rca() is a sync function that calls ask_llm_json_sync()
    # which bridges async→sync via ThreadPoolExecutor
    result = generate_rca(rca_input, ask_llm_json=agent.ask_llm_json_sync)

    state.rca_report = result.to_text()     # plain-text report
    state.rca_data   = result.model_dump()  # structured data for the HTML report
    await context.send_message(state)

# In BaseAgent — the async/sync bridge:
def ask_llm_json_sync(self, system_prompt, user_message) -> dict:
    with concurrent.futures.ThreadPoolExecutor(max_workers=1) as pool:
        future = pool.submit(
            asyncio.run,
            self.ask_llm_json(system_prompt, user_message),
        )
        return future.result()   # blocks until done

🔔 Executor 6 — notification_node

Terminal Executor

What it does

Plain English: The Notification node always runs last — whether blocking was found or not. It writes the permanent record to the database, generates human-readable reports, and emails the DBA team if something serious happened.

The key MAF concept here is yield_output(state). In LangGraph, the terminal node connects to a special END constant. In MAF, the terminal node signals completion by calling await context.yield_output(state) — this is what the workflow.run() call returns to main.py.

@executor(id="notification")
async def notification_node(state: BlockingState, context: WorkflowContext) -> None:
    cfg = get_config()

    if not state.has_blocking:
        await context.yield_output(state)  # clean cycle — nothing to log
        return

    errors = []

    # Write BlockingEventLog (permanent audit record)
    try:
        write_blocking_event(AuditLogInput(...), ...)
    except Exception as e:
        errors.append(str(e))     # collect error, don't crash

    # Render HTML + Markdown report
    try:
        html_path = render_reports(ReportInput(...))
        state.html_report_path = str(html_path)
    except Exception as e:
        errors.append(str(e))

    # Email DBA if kill was executed or approval is needed
    if state.dba_approval_required or state.kill_executed:
        try:
            send_dba_approval_email(DbaEmailInput(...))
        except Exception as e:
            errors.append(str(e))

    state.errors = errors

    # yield_output() — this is what AGENT_WORKFLOW.run() returns to main.py
    # In LangGraph this was add_edge("notification", END)
    await context.yield_output(state)

🛠️ The 13 Tools

All 13 tools live in src/tools/ and are identical to the LangGraph version. They contain zero LangChain or LangGraph imports — they are plain Python functions that take Pydantic input models and return Pydantic output models. This is why the MAF rewrite was safe: the hard part (the SQL logic) needed no changes at all.

detection.py

Queries sys.dm_exec_requests and sys.dm_exec_sessions to find all blocking chains. Returns the head blocker, all victims, wait times, and login names.

log_safety.py

Runs DBCC SQLPERF(LOGSPACE) to check how full the transaction log is. A nearly-full log means killing a session could trigger a large rollback that fills the log entirely.

locks.py

Queries sys.dm_tran_locks to identify the type of lock (shared/exclusive/update), the locked object (table/row/page), and whether lock escalation occurred.

plan_cache.py

Retrieves the execution plan XML for the blocking query. Checks for parallelism (CXPACKET waits), parameter sniffing, and missing index suggestions.

query_store.py

Reads the Query Store (SQL Server's built-in plan history) to detect plan regressions — cases where a query suddenly started using a worse execution plan.

scenario.py

Classifies blocking into one of 7 KB scenarios: 1=implicit transactions, 2=cursor, 3=orphaned transaction, 4=DTC, 5=rolling back, 6=idle open transaction, 7=parallel query.

sql_executor.py

Issues the actual KILL <spid> command. First re-verifies the SPID still exists. In dry-run mode, logs the intended action without executing it.

sql_validator.py

After a KILL, confirms the SPID is gone from sys.dm_exec_sessions. Implements Gap A (SPID recycling protection): checks that the same login still owns the session before declaring success.

audit_log.py

Writes to two permanent tables in AgentLogDB: BlockingEventLog (every blocking event) and KillAuditLog (every kill action). These records are append-only and immutable.

kill_rate.py

Counts kills in the past hour from KillAuditLog. Used by governance rule R10 to prevent the agent from killing too aggressively. Configurable via MAX_KILLS_PER_HOUR.

notifications.py

Sends email to the DBA team via SMTP. Used when a kill is executed (notification) or when DBA approval is required (R9: log nearly full). Configurable SMTP settings.

report_renderer.py

Generates an HTML and Markdown report for each blocking event. The report includes all diagnostic details, the kill decision, the RCA, and recommendations. Saved to the reports/ folder.

rca.py

Orchestrates the root-cause analysis. Queries historical recurrence data from AgentLogDB (has this blocker appeared before?), then calls the LLM via the ask_llm_json_sync bridge to generate recommendations.

🛡️ Governance Rules — Code-Enforced Safety

Plain English: These 8 rules are the agent's conscience. They are written in Python code inside _check_hard_gates() and the LLM cannot override them. If any rule matches, the decision is forced before the AI ever gets asked. This is what makes the system safe enough to run autonomously in production.

⚠️ The AI Never Overrides These Rules

Rules R2–R14 are checked in code before the LLM is called. If a rule fires, the cycle returns ALERT_ONLY or SKIP regardless of what the LLM would have decided. This is intentional — it keeps a human in the loop for edge cases.

RuleConditionForced DecisionWhy
R2 Wait time < KILL_THRESHOLD_MS (default 30s) SKIP Too short to be a real problem. Let it resolve naturally.
R3 SPID < 50 (SQL Server internal/system session) ALERT_ONLY Killing a system session could crash the server.
R9 Transaction log size > LOG_SIZE_KILL_THRESHOLD_GB ALERT_ONLY + DBA email Large rollback from a kill could fill the log and take the DB offline.
R10 Kills in the last hour ≥ MAX_KILLS_PER_HOUR ALERT_ONLY Rate limiter — prevents the agent from killing in a storm scenario.
R11 Victim logins don't match application_account_patterns SKIP Only kill blockers whose victims are known application accounts.
R12 Isolation level is in skip_isolation_levels list SKIP Some isolation levels (e.g. SERIALIZABLE) block intentionally.
R13 Session is already rolling back (scenario 5, percent_complete > 0, or ROLLBACK wait type) ALERT_ONLY Session is already fixing itself — killing it would restart the rollback and make things worse.
R14 Distributed transaction (DTC) wait type on the blocker ALERT_ONLY DTC transactions span multiple servers — killing one participant can corrupt data.

🔒 Security

DRY_RUN=true by Default

The most important safety feature. When DRY_RUN=true (the default), the agent never issues a real KILL command. It goes through every step — detection, analysis, determination — but the action node only logs "would have killed SPID X" and returns kill_status=DRY_RUN_SIMULATED. You must explicitly set DRY_RUN=false in the database config to enable live kills.

SPID Recycling Protection (Gap A)

SQL Server session IDs (SPIDs) can be recycled — SPID 99 might belong to batch_job_usr when the agent decides to kill it, but by the time the KILL executes, SPID 99 might belong to an innocent new connection. The sql_validator.py tool checks expected_login after the kill: if the SPID now shows a different login, it was recycled and the agent reports CONFIRMED_GONE (the original bad session is gone). This prevents accidentally killing innocent sessions.

Immutable Audit Trail

Every kill is recorded in KillAuditLog (append-only). Every blocking event is recorded in BlockingEventLog. These tables are never updated or deleted — they form a permanent audit trail that can be reviewed by DBAs, security teams, or compliance auditors.

Secrets Management

  • .env files are listed in .gitignore — never committed to git
  • API keys (LLM_API_KEY) support both explicit keys and DefaultAzureCredential (managed identity for cloud deployments)
  • SQL logins use least-privilege accounts — the monitor account has read-only DMV access; only the kill account can issue KILL

SQL Injection Prevention

All SQL queries in the tools use parameterized statements (cursor.execute(sql, params)), never string formatting. The only dynamic SQL is the KILL <spid> command — the SPID is an integer, validated to be > 50 (R3 gate) before use.

Lock Timeout Protection

Every SQL query in the agent runs after SET LOCK_TIMEOUT 30000 — if a DMV query itself gets blocked (which can happen during heavy contention), SQL Server raises an error after 30 seconds instead of waiting forever. This prevents the monitoring agent from being blocked by the very condition it's trying to detect.

⚠️ Error Handling

Tool-Level Isolation

Plain English: Each of the 6 diagnostic tools in detection_node is wrapped in a try/except. If the plan cache query fails (maybe that DMV is locked), the agent records the error but continues — it still runs the other 5 tools and makes the best decision it can with partial data. One failing tool doesn't crash the whole cycle.

# Pattern used in detection_node for every tool:
try:
    plan_result = analyze_plan_cache(...)
    state.plan_cache_hit = plan_result.plan_cache_hit
except Exception as e:
    state.errors.append(f"plan_cache: {e}")   # record it
    # continue — other tools still run

Cycle-Level Crash Recovery

In main(), every call to run_cycle() is wrapped in try/except. If the entire cycle crashes (e.g., the SQL Server is unreachable), the error is logged, but the polling loop continues. The agent will try again on the next interval.

LLM Timeout and Retry

The FoundryChatClient is configured with:

  • LLM_TIMEOUT_SECONDS (default 30) — if the GPT API doesn't respond in 30 seconds, the request is cancelled
  • LLM_MAX_RETRIES (default 2) — transient failures are retried automatically

Async/Sync Bridge Error Isolation

The ask_llm_json_sync() bridge runs the async LLM call in a dedicated thread via ThreadPoolExecutor. If the async call raises an exception, future.result() re-raises it in the calling thread, where it's caught by the tool-level try/except in rca_node.

Config DB Unavailability

If AgentConfigDB is unreachable at startup, load_config() falls back to environment variables entirely and logs a warning. The agent still runs with env-var defaults — it doesn't fail to start.

The routing after detection_node short-circuits to notification_node if has_blocking=False. No LLM executors are called. On a healthy SQL Server, the agent consumes no AI credits.

Singleton FoundryChatClient

BaseAgent._CLIENT is a class-level variable — the FoundryChatClient is created once and reused across all cycles. This avoids connection setup overhead on every cycle.

Fully Async Pipeline

All executor functions are async def. LLM calls use await — Python's event loop can handle other work while waiting for GPT responses. The sync bridge for rca.py uses a dedicated thread, keeping the main event loop unblocked.

Configurable Polling Interval

POLL_INTERVAL_SECONDS (default 45s, configurable in AgentConfigDB) controls how often cycles run. Faster = more responsive; slower = less SQL Server overhead.

get_config() Singleton (MAF pattern)

Unlike LangGraph which injected config via RunnableConfig["configurable"] into every node function, MAF has no built-in config injection. This codebase solves it with a module-level singleton:

# orchestrator/config.py
_ACTIVE_CONFIG: dict = {}        # populated once at startup

def get_config() -> dict:
    return _ACTIVE_CONFIG          # any executor can call this

def load_config() -> dict:
    global _ACTIVE_CONFIG
    # ... read .env and AgentConfigDB ...
    _ACTIVE_CONFIG = config
    return config
